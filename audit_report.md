# AI Engineering, Security, and Product Experience Audit Report

## A. Executive Summary

This report provides an in-depth audit of the "Engineering Copilot" project, a multi-layered application that provides a chat-based interface for RAG, QA, and coding tasks. The system is well-structured, with a clear separation of concerns between the Streamlit UI, the Java API gateway, the Python agent, and the MongoDB data store. The use of Docker Compose for orchestration and the detailed `README.md` file make the project easy to set up and run.

The Python agent is the core of the system, and it implements a sophisticated RAG pipeline with intent routing, context retrieval, and response validation. The use of a prompt registry and a structured `agent_runtime` shows a commitment to maintainability and quality.

However, there are several areas where the project can be improved. The most critical issue is the presence of hardcoded secrets in the `.env.example` file, which poses a significant security risk. Other high-priority recommendations include implementing response streaming across the entire stack to improve perceived performance, introducing correlation IDs for better observability, and improving the RAG pipeline with a more advanced reranking model.

Overall, the project is in good health, but there are several opportunities to improve its performance, security, and user experience. This report provides a prioritized list of recommendations to help the development team focus their efforts on the most impactful improvements.

### Top 5 Improvement Opportunities

1.  **Rotate hardcoded secrets in `.env.example`**: A critical security fix that is very easy to implement.
2.  **Implement response streaming from the Python agent to the Java API and then to the Streamlit UI**: This will drastically improve the perceived performance and user experience for long-running LLM tasks.
3.  **Introduce correlation IDs across all services**: This will enable distributed tracing and make it possible to debug requests as they flow through the system.
4.  **Implement a reranking stage with a more advanced model**: This will improve the quality of the context provided to the LLM, leading to more accurate and relevant answers.
5.  **Add integration tests that cover the interactions between the services**: This will increase confidence in the correctness of the system as a whole.

## B. Architecture Overview

The system is a multi-layered application designed to function as an "Engineering Copilot". It provides a chat-based interface for users to ask questions, generate code, and query databases, with a focus on Retrieval-Augmented Generation (RAG) from a local knowledge base.

The architecture consists of four main components, orchestrated by Docker Compose:

1.  **Streamlit UI (`streamlit-ui`)**: A Python-based web application that provides the user interface. It runs on port `8501` and communicates with the Java API gateway.

2.  **Java API Gateway (`java-api`)**: A Spring Boot application that serves as the backend-for-frontend (BFF) and a secure gateway. It exposes a REST API on port `8080`, handles user authentication (JWT-based), and proxies requests to the Python agent layer.

3.  **Python Agent (`python-agent`)**: A FastAPI application that contains the core AI and business logic. It runs on port `8000` and is responsible for:
    *   Intent routing (classifying user requests into QA, code generation, etc.).
    *   RAG pipeline execution (retrieving context from MongoDB).
    *   Prompt construction and LLM interaction with Ollama.
    *   Output validation and formatting.
    *   Interaction with the MCP server for web search.

4.  **MongoDB (`mongo`)**: A MongoDB database instance running on port `27017`, used as the primary data store for:
    *   Documents and chunks for the RAG pipeline.
    *   Embeddings for the chunks.
    *   Chat session history and execution traces.

5.  **Ollama (`ollama`)**: A service for running local LLMs and embedding models. It is conditionally included in the Docker Compose setup (for CPU or GPU) and runs on port `11434`. The Python agent communicates with Ollama to perform inference and generate embeddings.

### Data Flow

The typical data flow for a user query is as follows:

1.  The user interacts with the **Streamlit UI**, submitting a message.
2.  The UI sends a request to the **Java API Gateway**.
3.  The gateway validates the request and forwards it to the **Python Agent**.
4.  The Python agent's `router` classifies the user's intent (e.g., QA, CODE, SQL).
5.  The `rag` module retrieves relevant context (chunks) from **MongoDB** based on the user's query.
6.  The agent constructs a prompt using the user's query, retrieved context, and conversation history.
7.  The agent sends the prompt to the **Ollama** service to get a response from the configured LLM.
8.  The agent processes the LLM's response, validates it, and appends citations based on the retrieved context.
9.  The response is streamed back through the Java API Gateway to the Streamlit UI, which displays it to the user.

## C. Key Findings

### Performance

*   **N+1 Problem in LLM Instantiation**: The `general_llm()`, `code_llm()`, and `embedding_model()` functions in `python-agent/app/llm.py` use global singletons, which is good. However, they are called within the request lifecycle (`run_agent_with_trace`, `stream_agent_tokens`). While this is not a classic N+1 database problem, it does mean that the models are not pre-loaded at application startup, potentially adding latency to the first request.
*   **Blocking I/O in Streaming Endpoint**: The `chat_stream` endpoint in `python-agent/app/main.py` uses `iterate_in_threadpool` to wrap a synchronous generator. While this prevents the entire server process from blocking, it's less efficient than a fully asynchronous implementation. The underlying agent logic in `stream_agent_tokens` is also synchronous.
*   **No Caching for Embeddings**: The `retrieve_context` function in `python-agent/app/rag.py` re-computes the query embedding for every request. There is no caching for identical queries.
*   **No Caching for Web Search**: The `run_web_search` function in `python-agent/app/web.py` does not implement any caching for web search results. Repeated queries for the same information will result in redundant calls to the MCP server.
*   **Full Collection Scan for Chunks**: The `_get_cached_chunks` function in `python-agent/app/rag.py` loads all embedded chunks into memory from MongoDB. While it has a TTL-based cache, the initial population of this cache involves fetching all chunks, which could be slow and memory-intensive as the number of documents grows. The `_MAX_CACHED_CHUNKS` constant is a safeguard, but it's not a scalable solution.

### UI/UX

*   **No Visual Feedback for Long-Running Operations**: The Streamlit UI in `streamlit-ui/app.py` shows a simple "Generating response..." message. For long-running operations like RAG or complex code generation, there is no granular feedback to the user about what is happening (e.g., "Retrieving documents...", "Generating code...").
*   **No Streaming in Non-Streaming Mode**: When streaming is disabled, the UI waits for the full response before displaying anything, which can lead to a poor user experience for long-running tasks.
*   **Limited Error Handling**: The UI has basic error handling, but it often displays raw exception messages to the user (e.g., "Streaming failed: ...").
*   **No Way to View Trace Information**: The backend generates detailed trace information, but this is not exposed in the UI for debugging or for curious users.
*   **File Upload is a No-Op**: The file upload component in the Streamlit UI is present but not connected to any backend functionality.

### RAG and LLM quality

*   **Simple Reranking**: The `rerank_docs` function in `python-agent/app/rag.py` uses a simple weighted average of semantic similarity, keyword overlap, and a domain bonus. More advanced reranking models could provide better results.
*   **No Query Rewriting**: The RAG pipeline does not include a query rewriting step. For conversational queries, the user's last message might not be the best query for retrieval.
*   **Basic Groundedness Check**: The `check_groundedness` function in `python-agent/app/checker.py` uses token overlap to check for groundedness. This is a good start, but it can be brittle and may not catch more subtle hallucinations.
*   **No Feedback Loop**: There is no mechanism for users to provide feedback on the quality of the responses, which could be used to improve the system over time.

### API design

*   **Inconsistent DTOs**: The `ChatResponse` DTO in the Java API (`java-api/src/main/java/com/example/copilot/dto/ChatResponse.java`) has a `citations` field that is a `List<Map<String, Object>>`. This is not a strongly-typed DTO, which can lead to issues with serialization and client-side handling. The Python agent returns a more structured citation object.
*   **No API Versioning**: The API does not use versioning in the URL or headers.
*   **Missing OpenAPI Docs for Some Endpoints**: The `ChatController` has OpenAPI annotations, but other controllers (like `AuthController`) may be missing them.

### MCP web search

*   **Limited Error Handling**: The `McpWebSearchProvider` in `python-agent/app/mcp/client.py` has basic error handling, but it's not clear how it would handle different types of failures from the MCP server (e.g., timeouts, rate limits).
*   **No Fallback**: If the MCP server fails, there is no fallback mechanism.

### Security and privacy

*   **Hardcoded Secret in `.env.example`**: The `.env.example` file contains a default `JWT_SECRET` and `ADMIN_API_KEY`. While this is an example file, developers might copy it to `.env` without changing the secrets, leading to a security risk.
*   **Potentially Unsafe Deserialization in Streamlit UI**: The Streamlit UI's `stream_gateway_response` function uses `json.JSONDecoder().raw_decode()` to parse incoming JSON. While this is not inherently unsafe, it's a complex way to handle NDJSON and could be a source of errors if the input is not well-formed.
*   **No Input Sanitization on User Prompt**: The user's prompt is passed directly to the LLM. While the prompts are designed to be robust, there is no explicit sanitization to prevent prompt injection attacks.

### Reliability and error handling

*   **Limited Retries**: The `agent_runtime.py` has a retry loop for relevance, but there are no other retry mechanisms for transient failures (e.g., network errors when calling Ollama or the MCP server).
*   **Inconsistent Error Propagation**: It's not clear how errors from the Python agent are propagated to the Java API and then to the Streamlit UI. The UI seems to display raw exception messages in some cases.

### Observability

*   **Basic Tracing**: The `timed_node` function in `python-agent/app/tracing.py` provides basic performance tracing, but it's not integrated with a standard distributed tracing system like OpenTelemetry.
*   **No Metrics**: The application does not expose any Prometheus-style metrics for monitoring.
*   **No Centralized Logging**: Each service logs to its own console, but there is no centralized logging solution in place.

### Testing

*   **Missing Integration Tests**: There are unit tests for the Python agent and the Java API, but there are no integration tests that cover the interactions between the services.
*   **No E2E Tests**: There are no end-to-end tests that simulate a user interacting with the Streamlit UI and verify the entire workflow.
*   **No RAG Evaluation Tests**: While there is an evaluation harness (`python-agent/app/evals`), it's not clear how it's used or what it covers. There are no "golden" test sets for evaluating RAG quality.

### Documentation

*   **Good README**: The `README.md` file is well-written and provides a good overview of the project.
*   **Missing Architecture Documentation**: There is no detailed architecture documentation that explains the design decisions and the interactions between the components in more detail.
*   **Missing API Documentation**: While there is some OpenAPI documentation, it's not complete.

### Maintainability

*   **Monolithic Python Agent**: The `python-agent` is becoming a monolith, with a lot of logic packed into `agent_runtime.py`. This could be broken down into smaller, more focused modules.
*   **Configuration Sprawl**: Configuration is spread across `.env` files, `docker-compose.yml`, and the `config.py` file in the Python agent. This can make it difficult to manage configuration.

## E. Performance Improvement Plan

### Reducing Latency

*   **Pre-load Models at Startup**: Modify the `python-agent` to load the LLM and embedding models at application startup instead of on the first request. This can be done by calling `general_llm()`, `code_llm()`, and `embedding_model()` in the `startup` event handler in `python-agent/app/main.py`.
*   **Asynchronous Agent Runtime**: Refactor the `agent_runtime.py` and `main.py` to be fully asynchronous. This will involve using `async def` for the node functions and using an `async` compatible LLM library. This will significantly improve the performance of the streaming endpoint.
*   **Connection Pooling**: Ensure that the `WebClient` in the `AgentGatewayService` is configured to use a connection pool. This is the default behavior in Spring WebFlux, but it's worth verifying.

### Reducing LLM Calls

*   **Implement a Semantic Cache**: Before calling the LLM, check a cache for semantically similar queries that have already been answered. This can be implemented using a vector database to store query-response pairs.

### Reducing Token Usage

*   **Context Pruning**: Before sending the context to the LLM, prune it to only include the most relevant parts. This can be done by using a more advanced reranking model and only including the top-k chunks.
*   **Summarize Conversation History**: Instead of sending the full conversation history to the LLM, summarize it first. This will reduce the number of tokens used for conversation history.

### Improving Retrieval Speed

*   **Optimize MongoDB Queries**: Ensure that all queries to MongoDB are using indexes. The `ensure_indexes` function is a good start, but it's worth analyzing the query patterns to see if additional indexes are needed.
*   **Use a Dedicated Vector Database**: For larger-scale applications, consider using a dedicated vector database like Pinecone or Weaviate instead of relying on MongoDB for vector search. These databases are optimized for fast vector similarity search.

### Improving Streaming Behavior

*   **Full Asynchronous Pipeline**: As mentioned above, refactor the entire pipeline to be asynchronous. This will provide the best streaming performance.
*   **Stream Intermediate Steps**: In addition to streaming the final response, stream the intermediate steps to the UI (e.g., "Retrieving documents...", "Generating code..."). This will provide more feedback to the user.

### Improving Caching

*   **Cache Embeddings**: Implement a cache for query embeddings. A simple in-memory cache with a TTL or an LRU policy would be a good start.
*   **Cache Web Search Results**: Implement a cache for web search results. The cache key should be the search query.

### Improving Concurrent User Support

*   **Scalable Agent Service**: The Python agent is currently a single process. To support more concurrent users, it will need to be scaled horizontally. This can be done by running multiple instances of the agent behind a load balancer.
*   **Asynchronous Architecture**: A fully asynchronous architecture will allow the agent to handle more concurrent requests with fewer resources.

## F. UI/UX Improvement Plan

### Navigation

*   **Clearer Session Management**: The "New session" button is good, but it would be better to have a list of recent sessions that the user can switch between.
*   **Organize Sidebar**: The sidebar is getting crowded. Consider using expanders or tabs to organize the settings.

### Chat/Query Experience

*   **Rich Input Area**: The chat input is a simple text box. Consider a richer input area that supports markdown, code snippets, and file attachments.
*   **Autocomplete for Prompts**: For common prompts, provide autocomplete suggestions.

### Loading and Progress States

*   **Granular Progress Indicators**: As mentioned in the performance plan, provide more granular feedback to the user during long-running operations. For example, show a spinner with a message like "Retrieving documents...", "Generating code...", etc.
*   **Skeleton UI**: While waiting for the response, show a skeleton UI that mimics the structure of the final response. This will make the UI feel faster.

### Error Messages

*   **User-Friendly Error Messages**: Instead of displaying raw exception messages, show user-friendly error messages with a unique error ID. The user can then provide this ID to the support team for debugging.
*   **Actionable Errors**: When possible, provide actionable advice in the error message. For example, if the user's query is ambiguous, suggest ways to rephrase it.

### Source/Citation Presentation

*   **Interactive Citations**: Make the citations interactive. When the user clicks on a citation, show the full source document in a modal or a sidebar.
*   **Highlight Cited Text**: In the source document, highlight the specific text that was used to generate the response.

### File Upload Experience

*   **Connect File Upload to Backend**: Connect the file upload component to a backend endpoint that can ingest the files and add them to the RAG knowledge base.
*   **Show Upload Progress**: Show a progress bar while the files are being uploaded and processed.

### Feedback Collection

*   **Thumbs Up/Down**: Add a thumbs up/down button to each response. When the user clicks the button, ask for more detailed feedback.
*   **Feedback Form**: Provide a link to a feedback form where users can provide more detailed feedback.

### Result Formatting

*   **Collapsible Code Blocks**: For long code blocks, make them collapsible.
*   **Copy to Clipboard**: Add a "Copy to clipboard" button to each code block.

### Export/Share Workflows

*   **Export Conversation**: Allow the user to export the current conversation as a markdown or JSON file.
*   **Share Conversation**: Allow the user to share a link to the current conversation.

### Accessibility

*   **Keyboard Navigation**: Ensure that the entire UI can be navigated using the keyboard.
*   **Screen Reader Support**: Ensure that the UI is accessible to screen readers. Add ARIA labels to all interactive elements.

## G. RAG and Agent Quality Improvement Plan

### Chunking

*   **Semantic Chunking**: The current chunking strategy is not clear from the code, but it appears to be based on simple document splitting. Consider using a semantic chunking strategy that splits the documents based on the meaning of the text.
*   **Vary Chunk Size**: Experiment with different chunk sizes to find the optimal size for your documents and queries.

### Retrieval

*   **Hybrid Search**: Combine the current vector-based retrieval with a keyword-based search (like BM25). This will improve retrieval for queries that contain specific keywords.
*   **Query Rewriting**: Before retrieving documents, rewrite the user's query to be more suitable for retrieval. For example, for a conversational query like "Can you explain that in more detail?", you would want to rewrite it to include the context of the conversation.

### Reranking

*   **Cross-Encoder Reranking**: Use a cross-encoder model for reranking. Cross-encoders are more accurate than the simple weighted average used in the current implementation.
*   **Diversity in Reranking**: When reranking, consider the diversity of the results. You don't want to show the user multiple chunks that say the same thing.

### Prompt Design

*   **Prompt Templating Engine**: The current prompt registry is good, but it could be improved by using a more powerful templating engine like Jinja2. This would allow for more complex prompt logic.
*   **Prompt A/B Testing**: Implement a framework for A/B testing different prompts to see which ones perform better.

### Grounding

*   **More Robust Groundedness Check**: Use a more advanced model for the groundedness check. For example, you could use a natural language inference (NLI) model to check if the generated response is entailed by the retrieved context.
*   **Grounding with Confidence Score**: Instead of a binary "grounded" or "not grounded" flag, provide a confidence score for the groundedness of the response.

### Citations

*   **Chunk-Level Citations**: Instead of citing the entire document, cite the specific chunk that was used to generate the response.
*   **Highlight Cited Text**: As mentioned in the UI/UX plan, highlight the cited text in the source document.

### Evaluation

*   **Golden Dataset**: Create a "golden" dataset of queries and expected responses. This can be used to evaluate the performance of the RAG pipeline.
*   **Automated Evaluation**: Automate the evaluation process so that you can track the performance of the system over time. Use metrics like retrieval precision and recall, and LLM response quality.

### Tool Use

*   **Tool Use for RAG**: In addition to using tools for web search, consider using tools for other RAG-related tasks, such as query rewriting or summarization.
*   **Tool Failure Handling**: Improve the error handling for tool use. If a tool fails, the agent should be able to gracefully handle the failure and try a different approach.

### Conversation Memory

*   **Summarize Memory**: As mentioned in the performance plan, summarize the conversation memory before sending it to the LLM.
*   **Selective Memory**: Instead of sending the entire conversation history, only send the parts that are relevant to the current query.

### Coding-Task Safety

*   **Code Sandbox**: When the agent generates code, run it in a sandboxed environment to prevent it from accessing sensitive resources.
*   **Static Analysis**: Before running the code, perform static analysis to check for potential security vulnerabilities.

## I. Testing Gaps and Recommended Tests

### Missing Test Coverage

*   **Integration Tests**: There are no integration tests that cover the interactions between the services. For example, there are no tests that send a request to the Java API and verify that the Python agent is called correctly.
*   **End-to-End Tests**: There are no end-to-end tests that simulate a user interacting with the Streamlit UI and verify the entire workflow.
*   **RAG Evaluation Tests**: While there is an evaluation harness, it's not clear how it's used or what it covers. There are no "golden" test sets for evaluating RAG quality.
*   **API Contract Tests**: There are no tests that verify that the API contracts are not broken by changes to the DTOs or controller methods.
*   **UI Tests**: There are no automated tests for the Streamlit UI.
*   **MCP Tool Tests**: There are no tests for the MCP web search tool.

### Recommended Test Cases

*   **RAG Retrieval Correctness**:
    *   Given a specific query, assert that the correct documents are retrieved from MongoDB.
    *   Given a query with a specific domain, assert that the reranking logic correctly boosts the score of documents from that domain.
*   **LLM Response Grounding**:
    *   Given a query and a set of retrieved documents, assert that the generated response is grounded in the documents.
    *   Given a query where the retrieved documents are irrelevant, assert that the agent abstains from answering.
*   **Hallucination Checks**:
    *   Create a test set of queries where the answer is not in the knowledge base. Assert that the agent does not hallucinate an answer.
*   **API Validation**:
    *   Send invalid requests to the API and assert that the correct error responses are returned.
    *   Send requests with invalid JWTs and assert that the requests are rejected.
*   **MCP Search Failures**:
    *   Mock the MCP server to return an error and assert that the agent handles the error gracefully.
    *   Mock the MCP server to return no results and assert that the agent handles this case correctly.
*   **UI Session Behavior**:
    *   Assert that a new session is created when the "New session" button is clicked.
    *   Assert that the conversation history is cleared when a new session is created.
*   **File Upload Handling**:
    *   Once the file upload functionality is implemented, add tests to verify that files are uploaded and processed correctly.
*   **Timeout and Retry Scenarios**:
    *   Mock the LLM or MCP server to be slow and assert that the timeout and retry logic works as expected.
*   **Security Edge Cases**:
    *   Add tests for prompt injection attacks.
    *   Add tests for insecure file uploads.
*   **Multi-User Concurrency**:
    *   Add tests that simulate multiple users interacting with the system at the same time. Assert that the system remains stable and that there are no race conditions.

## J. Observability and Monitoring Plan

### Logs

*   **Structured Logging**: All services should use structured logging (e.g., JSON). This will make it easier to parse and analyze the logs.
*   **Correlation IDs**: As mentioned in the prioritized recommendations, introduce correlation IDs across all services. The correlation ID should be generated at the UI layer and passed through all subsequent service calls.
*   **Centralized Logging**: Use a centralized logging solution like the ELK stack (Elasticsearch, Logstash, Kibana) or Grafana Loki to collect and analyze logs from all services.

### Metrics

*   **Prometheus Metrics**: All services should expose Prometheus-style metrics.
*   **Key Metrics to Track**:
    *   **API Gateway**: Request latency, request rate, error rate.
    *   **Python Agent**: Request latency, request rate, error rate, LLM call latency, RAG retrieval latency, tool use latency.
    *   **LLM**: Token usage, cost per request.
    *   **RAG**: Retrieval precision and recall.
    *   **UI**: Page load time, time to first interaction.

### Traces

*   **Distributed Tracing**: Use a distributed tracing system like OpenTelemetry to trace requests as they flow through the system.
*   **Trace Visualization**: Use a tool like Jaeger or Zipkin to visualize the traces.

### Dashboards

*   **Create Grafana Dashboards**: Create Grafana dashboards to visualize the metrics and logs.
*   **Key Dashboards**:
    *   **System Overview**: A high-level overview of the health of the system.
    *   **API Gateway**: A detailed view of the performance of the API gateway.
    *   **Python Agent**: A detailed view of the performance of the Python agent.
    *   **RAG Performance**: A dashboard to track the performance of the RAG pipeline.
    *   **LLM Usage**: A dashboard to track LLM token usage and cost.

### Alerting

*   **Alert on Key Metrics**: Set up alerts for key metrics, such as high error rates, high latency, and low RAG retrieval precision.
*   **Alerting Channels**: Send alerts to a channel that the development team actively monitors, such as Slack or PagerDuty.

### User Feedback

*   **Track User Feedback**: As mentioned in the UI/UX plan, track user feedback (e.g., thumbs up/down).
*   **Feedback Dashboard**: Create a dashboard to visualize user feedback. This will help you identify areas where the system can be improved.

## K. Quick Wins

*   **Rotate hardcoded secrets in `.env.example`**: This is a critical security fix that is very easy to implement.
*   **Add a "Copy to clipboard" button to code blocks in the Streamlit UI**: This is a small UI improvement that will significantly improve the user experience for developers.
*   **Cache embeddings for identical queries**: This is a low-effort performance improvement that can provide a significant reduction in latency for repeated queries.
*   **Use strongly-typed DTOs for API responses**: This is a low-effort change that will improve the developer experience and reduce the risk of serialization errors.
*   **Pre-load models at startup**: This is a simple change that will reduce the latency of the first request to the Python agent.

## L. Longer-Term Roadmap

*   **Full Asynchronous Pipeline**: Refactor the entire pipeline to be asynchronous. This will provide the best performance and scalability.
*   **Advanced RAG Pipeline**: Implement a more advanced RAG pipeline with features like query rewriting, hybrid search, and cross-encoder reranking.
*   **Comprehensive Evaluation Framework**: Build out a comprehensive evaluation framework with a golden dataset and automated evaluation.
*   **Multi-Tenant Architecture**: If the application needs to support multiple users with different knowledge bases, it will need to be refactored to support multi-tenancy.
*   **CI/CD for Machine Learning (MLOps)**: Implement a CI/CD pipeline for the machine learning components of the system. This will include automated training, evaluation, and deployment of the RAG and LLM models.
*   **Self-Service Document Ingestion**: Allow users to upload and manage their own documents through the UI.
*   **Fine-tuning LLMs**: For very specific domains, consider fine-tuning the LLMs on your own data.

## M. Open Questions

*   **Chunking Strategy**: The chunking strategy is not clear from the code. How are the documents in the `books/` directory chunked? What is the chunk size and overlap?
*   **Evaluation Harness**: The evaluation harness is present, but it's not clear how it's used. What metrics are being tracked? What is the process for running evaluations?
*   **MCP Server Implementation**: The MCP server is used for web search, but the implementation of the server itself is not in the repository. How is the server implemented? What search provider is it using?
*   **Deployment Environment**: The `README.md` file mentions different deployment modes, but it's not clear what the target deployment environment is. Is it a single server, a Kubernetes cluster, or a serverless environment?
